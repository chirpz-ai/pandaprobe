Evaluation **monitors** let you automate recurring evaluations. Instead of manually creating eval runs, a monitor persists your evaluation configuration and automatically spawns eval runs on a schedule.

## Creating a monitor

```bash
POST /api/v1/evaluations/monitors
```

```json
{
  "name": "Daily production trace eval",
  "target_type": "TRACE",
  "metrics": ["task_completion", "tool_correctness", "confidence"],
  "filters": {
    "status": "COMPLETED",
    "tags": ["production"]
  },
  "cadence": "daily",
  "sampling_rate": 0.3,
  "model": "openai/gpt-4o",
  "only_if_changed": true
}
```

### Request fields

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | Yes | Human-readable label (e.g., "Daily prod eval") |
| `target_type` | string | Yes | `"TRACE"` or `"SESSION"` |
| `metrics` | string[] | Yes | Metric names to run on each scheduled eval |
| `filters` | object | No | Scope the data the monitor evaluates (default: `{}` matches everything) |
| `cadence` | string | Yes | Firing schedule (see [Cadence options](#cadence-options) below) |
| `sampling_rate` | float | No | Fraction of matching data to evaluate per run (default: 1.0) |
| `model` | string | No | LLM model override for judge calls |
| `only_if_changed` | boolean | No | Skip run if no new data since last run (default: `true`) |
| `signal_weights` | object | No | Override signal weights (SESSION monitors only) |

### Filters by target type

<Tabs>
  <Tab title="TRACE monitors">
    | Filter | Type | Description |
    |---|---|---|
    | `date_from` | ISO 8601 | Traces started on or after |
    | `date_to` | ISO 8601 | Traces started before (exclusive) |
    | `status` | string | `PENDING`, `RUNNING`, `COMPLETED`, or `ERROR` |
    | `session_id` | string | Exact session ID |
    | `user_id` | string | Exact user ID |
    | `tags` | string[] | Traces matching ANY tag |
    | `name` | string | Substring match (case-insensitive) |
  </Tab>
  <Tab title="SESSION monitors">
    | Filter | Type | Description |
    |---|---|---|
    | `date_from` | ISO 8601 | Sessions from this time |
    | `date_to` | ISO 8601 | Sessions before this time |
    | `user_id` | string | Exact user ID |
    | `has_error` | boolean | Only sessions with/without errors |
    | `tags` | string[] | Traces matching ANY tag |
    | `min_trace_count` | int | Minimum traces in session (≥1) |
  </Tab>
</Tabs>

## Cadence options

Monitors support both predefined intervals and custom cron expressions.

### Predefined intervals

| Value | Schedule |
|---|---|
| `every_6h` | Every 6 hours |
| `daily` | Once per day |
| `weekly` | Once per week |

### Custom cron

Use `cron:<5-part expression>` for fine-grained control. The five parts follow standard cron syntax: `minute hour day-of-month month day-of-week`.

| Example | Schedule |
|---|---|
| `cron:0 3 * * *` | Daily at 3:00 AM UTC |
| `cron:0 6 * * 1-5` | Weekdays at 6:00 AM UTC |
| `cron:0 */4 * * *` | Every 4 hours |
| `cron:0 9 1 * *` | First of every month at 9:00 AM UTC |
| `cron:30 2 * * 0` | Sundays at 2:30 AM UTC |

## The `only_if_changed` flag

When `only_if_changed` is `true` (the default), the scheduled run is **skipped** if no new traces or sessions have arrived since the last run. This prevents wasting LLM costs on re-evaluating the same data.

Set to `false` if you want to run on schedule regardless — for example, if you've changed your LLM model or threshold and want to re-evaluate existing data.

## Monitor lifecycle

Monitors have two states:

| Status | Description |
|---|---|
| `ACTIVE` | Running on schedule, will spawn eval runs at each cadence tick |
| `PAUSED` | Schedule is suspended, no new runs will be created |

### Pause a monitor

```bash
POST /api/v1/evaluations/monitors/{monitor_id}/pause
```

Idempotent — safe to call on an already-paused monitor.

### Resume a monitor

```bash
POST /api/v1/evaluations/monitors/{monitor_id}/resume
```

Idempotent — safe to call on an already-active monitor.

## Trigger a run manually

Force an immediate eval run from a monitor, ignoring its cadence schedule:

```bash
POST /api/v1/evaluations/monitors/{monitor_id}/trigger
```

This creates a new eval run using the monitor's saved configuration (metrics, filters, sampling rate, model) and returns the run details. Useful for testing a monitor's configuration or triggering an ad-hoc evaluation.

**Rate limit:** 10/min

## Managing monitors

### List all monitors

```bash
GET /api/v1/evaluations/monitors?status=ACTIVE&limit=50
```

| Parameter | Description |
|---|---|
| `status` | Filter by `ACTIVE` or `PAUSED` |
| `limit` / `offset` | Pagination |

### Get monitor details

```bash
GET /api/v1/evaluations/monitors/{monitor_id}
```

Returns full configuration including `last_run_at`, `last_run_id`, and `next_run_at`.

**Response:**

```json
{
  "id": "a1b2c3d4-...",
  "project_id": "...",
  "name": "Daily production trace eval",
  "target_type": "TRACE",
  "metric_names": ["task_completion", "tool_correctness", "confidence"],
  "filters": {"status": "COMPLETED", "tags": ["production"]},
  "sampling_rate": 0.3,
  "model": "openai/gpt-4o",
  "cadence": "daily",
  "only_if_changed": true,
  "status": "ACTIVE",
  "last_run_at": "2026-03-28T03:00:00Z",
  "last_run_id": "e5f6g7h8-...",
  "next_run_at": "2026-03-29T03:00:00Z",
  "created_at": "2026-03-15T10:00:00Z",
  "updated_at": "2026-03-28T03:00:00Z"
}
```

### Update a monitor

```bash
PATCH /api/v1/evaluations/monitors/{monitor_id}
```

Partial update — only provided fields are changed:

```json
{
  "cadence": "cron:0 6 * * 1-5",
  "sampling_rate": 0.5,
  "metrics": ["task_completion", "tool_correctness", "confidence", "step_efficiency"]
}
```

### Delete a monitor

```bash
DELETE /api/v1/evaluations/monitors/{monitor_id}
```

Deletes the monitor configuration. Eval runs previously spawned by this monitor are preserved.

### List runs spawned by a monitor

```bash
GET /api/v1/evaluations/monitors/{monitor_id}/runs?limit=20
```

Returns the history of eval runs created by this monitor, ordered by creation time.

## Example: complete monitoring setup

Here's a typical setup for a production environment with both trace and session monitoring:

### 1. Daily trace evaluation

```bash
curl -X POST https://api.pandaprobe.com/api/v1/evaluations/monitors \
  -H "Authorization: Bearer $TOKEN" \
  -H "X-Project-ID: $PROJECT_ID" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Daily trace quality check",
    "target_type": "TRACE",
    "metrics": ["task_completion", "tool_correctness", "argument_correctness"],
    "filters": {"status": "COMPLETED", "tags": ["production"]},
    "cadence": "daily",
    "sampling_rate": 0.2,
    "only_if_changed": true
  }'
```

### 2. Weekly session evaluation

```bash
curl -X POST https://api.pandaprobe.com/api/v1/evaluations/monitors \
  -H "Authorization: Bearer $TOKEN" \
  -H "X-Project-ID: $PROJECT_ID" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Weekly agent reliability audit",
    "target_type": "SESSION",
    "metrics": ["agent_reliability", "agent_consistency"],
    "filters": {"min_trace_count": 3},
    "cadence": "weekly",
    "sampling_rate": 1.0,
    "signal_weights": {"confidence": 1.0, "loop_detection": 1.5},
    "only_if_changed": true
  }'
```

### 3. Monitor the monitors

```bash
curl https://api.pandaprobe.com/api/v1/evaluations/monitors \
  -H "Authorization: Bearer $TOKEN" \
  -H "X-Project-ID: $PROJECT_ID"
```

Check `last_run_at` and `next_run_at` to verify monitors are firing as expected.
